# session-start

**Точка входа в каждую новую сессию.** Решает: продолжить со старыми данными или настроить заново. Заменяет «всегда спрашивать имя заново».

## ⚡ Правило: всегда первый пиши в новой сессии

Агент **обязан** писать первым при старте сессии, не дожидаясь «привет» от пользователя. Сценарий выбирается по `setup_status` + `user_key`:

| Сценарий | Когда | Шаблон |
|---|---|---|
| **A** (`new`) | Нет профиля | Приветствие + меню «Экзамен / Олимпиада / Тема» (имя — в том же ответе) |
| **B** (`complete`) | Профиль готов | С возвращением + **меню Продолжить / Настроить заново** |
| **C** (`in_progress`) | Оборвалась настройка | Шаг N + **Продолжить настройку / Начать заново** |
| **D** (`owner`) | `user_key === owner` | Management-дашборд (drift / proposals / tests) |

Скрипт `scripts/session-start.mjs` выдаёт поле `proactive_message` с `template` + `context`. Агент подставляет плейсхолдеры и пишет первым сообщением.

## Триггер
- Новая сессия: `/new`, рестарт чата, первое сообщение, пауза > 2ч
- Срабатывает **раньше** `hello-intro` и любых других скиллов

## Скрипт (обязательно)
```bash
node scripts/session-start.mjs --user <user_key> [--telegram-id <id>] [--username @x]
node scripts/session-start.mjs --owner    # для webchat владельца (user_key=owner)
```
При наличии `telegram_id` скрипт автоматически резолвит pending-инвайты (`team-tasks`). Если `team_invites.count > 0` — сообщить пользователю, в какие команды он вступил.

## Логика

1. **Определить `user_key`** (см. `user-profile`): Telegram id → `tg-<id>`, иначе `local`/`<channel>-<id>`.
2. **Проверить `users/<user_key>/profile.md`:**

### A. Профиль есть и `setup_status: complete`
Поприветствовать по имени и **предложить выбор** (не загружать молча):

```
🌅 С возвращением, **<name>**!

Помню твою настройку: <purpose>, <subject/topic>, дедлайн <deadline>, <hours_per_week> ч/нед.

1. **Продолжить** — загружу твои данные и план, идём дальше
2. **Настроить заново** — начнём с чистого листа (старое сохраню в архив)

Цифра или слово.
```

- **«Продолжить» / 1 / «загрузи»** → прочитать `profile.md`, `plan.md`, `progress.md`, `tasks.md`. Кратко показать статус (ближайшие задачи / на какой неделе плана) → перейти в рабочий режим.
- **«Настроить заново» / 2 / «с нуля»** → **подтвердить** («Точно? Старый профиль уйдёт в архив»). При «да» — переименовать папку в `users/<user_key>/archive-<timestamp>/`, начать `hello-intro`.

### B. Профиль есть, но `setup_status: in_progress`
```
🌅 Привет снова, **<name>**! Мы не закончили настройку.

1. **Продолжить настройку** — с того места, где остановились (<следующий шаг>)
2. **Начать заново**
```
- Продолжить → возобновить онбординг с незаполненного шага.
- Заново → архив + `hello-intro`.

### C. Профиля нет (`new`)
Новый пользователь — **сразу онбординг** (`hello-intro`: приветствие + цель; имя в том же сообщении), без меню load/reset.

## Что НЕ делает
- ❌ Не загружает данные молча — всегда даёт выбор возвращающемуся пользователю.
- ❌ Не запускает рабочие скиллы (план/задачи/напоминания), пока не выбран режим и `setup_status` не `complete`.
- ❌ Не смешивает данные разных `user_key`.
- ❌ Не удаляет старый профиль без подтверждения — только в архив.

## После выбора
- **Загрузка завершена** → рабочий режим (см. `study-plan`, `daily-plan`, `tasks`, `checkins`).
- **Новый / заново** → онбординг: `hello-intro` → `purpose-select` → ветка → `setup-finalize`.

## Зависимости
- Читает данные через `user-profile`.
- Перед `hello-intro` (для нового/reset) или перед рабочим режимом (для load).

## Где живёт реальное исполнение
**`SOUL.md` → секция «Старт сессии»** — то, что агент реально выполняет. Этот скилл — дизайн-документ.
