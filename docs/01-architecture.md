# 01. Архитектура

## Что это

**Golden Hour (Золотой час)** — ИИ-агент-планировщик подготовки к экзаменам/олимпиадам/темам. Работает как Telegram-бот поверх платформы **OpenClaw** (агентный рантайм: gateway, каналы, сессии, cron, sub-агенты). Ядро агента — не программа, а набор markdown-инструкций, которые LLM читает в каждой сессии; детерминированная логика вынесена в Node-скрипты.

## Три слоя

```
┌─────────────────────────────────────────────────────────────┐
│  1. СЛОЙ ИНСТРУКЦИЙ (LLM читает при старте сессии)            │
│     SOUL.md · AGENTS.md · IDENTITY.md · skills/*/SKILL.md     │
│     → определяет поведение, фазы, стиль, безопасность         │
└─────────────────────────────────────────────────────────────┘
                          │ агент вызывает
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  2. СЛОЙ ДЕТЕРМИНИРОВАННОЙ ЛОГИКИ (Node ≥18, без LLM)         │
│     scripts/*.mjs → читают/пишут users/<key>/, печатают JSON  │
│     scripts/lib/*.mjs → общие модули (веса, план, даты...)    │
└─────────────────────────────────────────────────────────────┘
                          │ читает/пишет
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  3. СЛОЙ ДАННЫХ (файлы на диске, по пользователю)            │
│     users/<user_key>/ : profile.md, plan.md, progress.md,     │
│     tasks.yaml, plans/YYYY-MM-DD.json, timer/, materials/...  │
│     data/exam-topics/ : общие кодификаторы (read-only)        │
└─────────────────────────────────────────────────────────────┘
```

**Внешние интерфейсы** поверх этих слоёв:
- **Telegram-бот** — основной канал (через OpenClaw bindings).
- **Cron / heartbeat** — детерминированная рассылка напоминаний (`cron-deliver.mjs` → Telegram Bot API, без LLM).
- **Dashboard / Student Portal / Telegram Mini App** — веб-интерфейс (Python + PowerShell), читает/пишет те же `users/<key>/plans/*.json`.
- **Google Calendar** — двусторонняя синхронизация (`gcal.mjs`).
- **Sub-агенты** — `code-writer` (код по запросу) и `web-material-finder` (поиск материалов).

## Почему такое разделение

- **Детерминизм и повторяемость.** Веса задач, баланс дня, интервалы повторения, статистику нельзя доверять «арифметике в уме» LLM — они вычисляются скриптами и возвращаются как JSON. Агент только показывает `summary`.
- **Cron без затрат на модель.** Напоминания идут по расписанию скриптами; LLM не вызывается, доставка идемпотентна (state-файлы).
- **Контракт `{ ok: true, ... }` / `{ ok: false, error }`.** Каждый скрипт печатает один JSON в stdout. При `ok:false` агент не выдумывает результат.

## Главный цикл подготовки

Всё крутится вокруг папки `users/<user_key>/`. Скиллы — звенья одной цепи; следующее звено агент вызывает сам.

```
profile.md (настройка / онбординг)
   └─ study-plan ───────────────► plan.md (макро-план по приоритетам)
        └─ daily-plan ──(task-weighting + daily-balancer + recurring + spaced-repetition)──► plans/YYYY-MM-DD.json
             └─ checkins ──► утренний бриф + пинги задач + вечерний чек-ин + рефлексия
                  └─ timer ──► timer/ ──► зачёт задачи
                       └─ progress.md (+ streak, SR-интервалы)
                            └─ longterm-stats ──► статистика за период
```

Сквозные сигналы:
- **Единый источник правды — `profile.md`.** Уровни, приоритеты, нагрузка (`daily_load`), тема оформления живут там; все скиллы читают оттуда.
- **Слабые темы буст-ятся автоматически.** Низкий уровень → выше `eff_priority` → чаще в дневном плане → чаще в spaced repetition.
- **Temporal KG пополняется автоматически** из `checkin-record`, `plan-task` (done/start/skip), `timer credit` — не дублируется вручную.

## Жизненный цикл сессии

```
Новое сообщение / /new / рестарт
   │
   ▼
session-start.mjs --user <key>   →   JSON { setup_status, action, ... }
   │
   ├─ setup_status = new         → онбординг (skills/onboarding)
   ├─ setup_status = in_progress → продолжить онбординг с шага обрыва
   ├─ setup_status = complete    → меню «Продолжить / Настроить заново» → рабочий режим
   └─ user_key = owner           → management-режим (owner-profile), без онбординга
```

## Определение пользователя (`user_key`)

`user_key` берётся **только из метаданных канала** (`sender_id`/`chat_id`), никогда из текста сообщения.

| Источник | `user_key` |
|---|---|
| Telegram | `tg-<id>` (напр. `tg-1234567890`) |
| Другой канал | `<channel>-<id>` |
| Webchat / локально | `local` |
| Владелец воркспейса | `owner` |

## Технологический стек

| Область | Технология |
|---|---|
| Рантайм агента | OpenClaw (gateway на loopback, каналы, cron, sessions) |
| Скрипты | Node.js ≥18 (ESM `.mjs`); `npm install` только для опционального SQLite (`GH_USE_DB=1`) |
| Хранилище | Файлы (Markdown + YAML + JSONL + JSON); опционально SQLite |
| Дашборд/портал | Python (HTTP-серверы), Vanilla JS SPA, PowerShell (запуск/watchdog на Windows) |
| Рендер карточек | HTML + Chrome/Edge headless (`skills/study-cards/`) |
| Модель (сервер) | MiniMax-M3 (Anthropic-совместимый API) + локальный Ollama-фолбэк |
| Мониторинг | Grafana + Prometheus (опционально) |
| CI | GitHub Actions (unit-тесты + smoke) |
| Деплой | GitHub Actions → SSH ForceCommand → systemd |

Детали каждого слоя — в соответствующих разделах документации.
